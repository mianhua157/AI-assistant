# RAG 系统构建流程说明

## 整体架构

```
用户问题 → 检索 → 生成回答
           ↑
    向量知识库 (FAISS)
           ↑
    PDF + Wiki 文档
```

---

## 构建流程

### 第一步：build_wiki.py - 生成概念知识库

**目的**：从原始 PDF 中提取核心概念，生成结构化的 markdown 知识页

**输入**：
- `classification.pdf`（原始课程讲义）

**处理**：
1. 读取 PDF 全文
2. 对每个预定义概念（classification, logistic regression, knn, decision tree, naive bayes, lda, svm）
3. 调用 LLM 生成结构化 markdown 页面

**输出**：
- `wiki/classification.md`
- `wiki/logistic_regression.md`
- `wiki/k-nearest_neighbor.md`
- `wiki/decision_tree.md`
- `wiki/naive_bayes.md`
- `wiki/linear_discriminant_analysis.md`
- `wiki/support_vector_machine.md`

**意义**：这是第二层知识库，将分散在 PDF 中的概念整理成结构化页面

---

### 第二步：build_vectorstore.py - 构建向量数据库

**目的**：将 PDF 原始文档和 Wiki 概念页统一向量化存储

**输入**：
- `raw/*.pdf` - 原始 PDF 文档
- `wiki/*.md` - 生成的概念知识页

**处理**：
1. 加载 PDF 文档（标记 type="raw_pdf"）
2. 加载 Wiki markdown（标记 type="wiki"）
3. 合并两类文档
4. 切分为 chunk（800 字符/块，重叠 150）
5. 使用 embedding 模型生成向量
6. 保存到 FAISS 索引

**输出**：
- `faiss_index/index.faiss` - 向量索引
- `faiss_index/index.pkl` - 文档存储

**意义**：混合知识库，同时包含原始资料和结构化概念

---

### 第三步：rag.py - 检索与生成

**目的**：用户提问时，从向量库检索并生成回答

**流程**：
1. 用户输入问题
2. 生成问题的 embedding
3. 在 FAISS 中检索最相似的 chunk
4. 将检索结果 + 问题输入 LLM
5. 生成最终回答

---


---

## 关键改进

1. **双层知识结构**：原始 PDF + 结构化 Wiki
2. **元数据标记**：每个文档标记来源类型，便于追踪
3. **统一向量化**：PDF 和 Wiki 使用相同 embedding，可混合检索

## 运行顺序

```bash
# 1. 生成 wiki 概念页
python build_wiki.py

# 2. 构建向量库
python build_vectorstore.py

# 3. 启动应用
streamlit run app.py
```

### 第四阶段：简化 app.py（2026-04-20）

**目标**：按照图片要求，简化 `app.py`，正确显示 wiki 来源

**修改文件**：
- `app.py` - 重构为简洁版本

**主要改动**：
1. 移除不必要的导入（`os`, `re`）
2. 移除缓存机制（简化代码）
3. 移除关键词高亮功能（简化代码）
4. 使用 `@st.cache_resource` 缓存向量库加载
5. 直接显示来源类型和文件路径

**修改后代码**：
```python
import streamlit as st
from rag import load_vectorstore, ask_rag

# 加载向量库（带缓存）
@st.cache_resource
def get_vectorstore():
    return load_vectorstore()

vectorstore = get_vectorstore()

# 输入框
question = st.text_input("请输入你的问题：")

if question:
    with st.spinner("正在检索资料并生成回答..."):
        answer, docs = ask_rag(question, vectorstore)

    # 显示回答
    st.subheader("回答")
    st.write(answer)

    # 显示检索到的来源
    with st.expander("📚 检索到的来源"):
        for i, doc in enumerate(docs):
            st.markdown(f"**来源 {i+1}**")
            st.write("类型：", doc.metadata.get("type", "raw"))
            st.write("文件：", doc.metadata.get("source", "unknown"))
            st.write(doc.page_content[:500])
            st.markdown("---")
```

**代码简化对比**：
| 功能 | 修改前 | 修改后 |
|------|--------|--------|
| 代码行数 | 114 行 | 45 行 |
| 缓存机制 | Session State | 移除 |
| 关键词高亮 | 有 | 移除 |
| 来源显示 | 复杂格式化 | 简洁显示 |
| 向量库加载 | 全局变量 | `@st.cache_resource` |

### 第六阶段：优化 Prompt（2026-04-20）

**目标**：优化 RAG 生成的 prompt，让回答质量更高

**修改文件**：
- `rag.py` - `ask_rag()` 函数中的 prompt

**修改内容**：
```python
# 修改前
prompt = f"""你是一个机器学习助教，请根据提供的资料回答问题。

规则：
1. 优先使用资料内容
2. 如果资料中有相关内容，可以做合理归纳
3. 全部使用中文回答
4. 在答案末尾标注参考来源编号，如 [1][2]
5. 只有在资料明显无关时，才回答"我不知道"
...
"""

# 修改后
prompt = f"""你是一个机器学习助教，请根据资料回答问题。

要求：
1. 必须用自己的话总结，不要直接拼接原文
2. 优先使用 wiki 内容进行整理
3. 回答要结构清晰（可以分点）
4. 必须用中文
5. 在最后标注来源编号
...
"""
```

**关键改进**：
| 修改点 | 修改前 | 修改后 |
|--------|--------|--------|
| 总结要求 | 无明确要求 | 必须用自己的话总结 |
| Wiki 优先级 | 无 | 优先使用 wiki 内容 |
| 结构要求 | 无 | 结构清晰（可以分点） |
| 拒答机制 | 有（资料无关时） | 移除 |

**预期效果**：
1. 减少直接拼接原文的情况
2. Wiki 结构化知识被优先使用，回答更准确
3. 分点回答更清晰易读
4. 移除拒答机制，减少"我不知道"的情况

### 第七阶段：提高 Wiki 权重（2026-04-20）

**目标**：让 RAG 检索优先使用 Wiki 内容

**方法 1：提高 Wiki 权重（检索层）**

**修改文件**：
- `rag.py` - `ask_rag()` 函数

**修改内容**：
```python
# 修改前
docs = vectorstore.similarity_search(question, k=5)

# 修改后
# 先检索多一些（k=8）
docs = vectorstore.similarity_search(question, k=8)

# 按类型分离
wiki_docs = [d for d in docs if d.metadata.get("type") == "wiki"]
raw_docs = [d for d in docs if d.metadata.get("type") != "wiki"]

# 重新组合：wiki 前 2 个 + raw 前 3 个
docs = wiki_docs[:2] + raw_docs[:3]
```

**方法 2：只改 Prompt**

在 prompt 中添加：
```
要求：
1. 优先使用 wiki 类型的资料
2. 如果 wiki 有内容，尽量基于 wiki 组织答案
```

**关键改进**：
| 方法 | 修改位置 | 效果 |
|------|----------|------|
| 方法 1 | 检索层 | 强制保证 wiki 文档占比 |
| 方法 2 | Prompt | 引导模型优先使用 wiki |
